# 1. Produktidee / Grundphilosophie

## Ausgangslage

Aktuell existieren beim Tennisverein mehrere voneinander getrennte Systeme:

### NetXP Verein
Wird derzeit für klassische Vereinsverwaltung genutzt:
- Mitgliederdaten
- Buchhaltung
- Rechnungen
- Lastschrift / Abbuchungen

Mit NetXP besteht Unzufriedenheit. Ein späterer Austausch gegen ein besseres Tool ist denkbar.

### eBuSy
Wird für operative Buchungen genutzt:
- Platzbuchungen
- Hallenbuchungen
- Gruppentrainings

Positiv:
- eBuSy ist funktional gut
- eBuSy bietet eine gute API

Negativ:
- Frontend und Backend sind an manchen Stellen nicht informativ genug

### Website
- aktuelle Website: statische WordPress-Seite

---

## Zielbild

Es soll eine eigene Anwendung entstehen, die Informationen aus mehreren Quellen bündelt und später möglicherweise Teilbereiche ganz selbst übernimmt.

Kernidee:

> Eine eigene Integrations-, Analyse- und Anwendungsschicht über NetXP (oder späterem Ersatz) und eBuSy.

Diese eigene Anwendung soll:
- Daten aus bestehenden Systemen zusammenführen
- eigene zusätzliche Logik und Datenmodelle erlauben
- bessere Admin-Oberflächen bereitstellen
- bessere Informationen für Mitglieder liefern
- Frontend-Endpunkte für Website oder App bereitstellen

---

## Wichtiger Architektur-Grundsatz

**Nicht sofort NetXP und eBuSy komplett ersetzen.**

Sondern zunächst:

### Phase 1: Overlay / Integrationsschicht
Das eigene System liegt „über“ den bestehenden Systemen.

Es:
- liest Daten ein
- speichert ggf. eigene ergänzende Daten
- stellt bessere Ansichten und Workflows bereit

### Phase 2: Eigene Workflows
Beispiele:
- Arbeitsstunden digital erfassen
- Freigabeprozesse für Admins
- Traineransichten für Gruppenteilnehmer
- Auswertungen und Reports
- bessere Mitgliedersichten

### Phase 3: Eigene Source of Truth für Teilbereiche
Später können bestimmte Bereiche im eigenen System führend werden.

Beispiele:
- Arbeitsstunden
- Mitgliedsanträge
- interne Rollen / Status
- Kurs- und Gruppentraining-Logik

### Phase 4: Eventueller Teil- oder Vollersatz bestehender Tools
Erst sehr viel später denkbar:
- NetXP teilweise oder ganz ersetzen
- eBuSy in bestimmten Teilbereichen ergänzen oder teilweise ablösen

---

## Was als erstes gute MVP-Bausteine sind

Zwei frühe Use Cases wurden als besonders sinnvoll identifiziert:

### 1. Arbeitsstunden-Modul
Jedes Mitglied muss 5 Arbeitsstunden pro Jahr ableisten.

Ziel:
- Stunden digital erfassen
- Mitglieder sehen ihren Stand online
- Admins können Stunden freigeben / bestätigen

Warum geeignet:
- klarer Prozess
- klarer Nutzen
- wenig Risiko
- guter erster Bereich für eigene Logik

### 2. Traineransicht für Gruppentrainings
Trainer sollen genau sehen können, wer welche Gruppentrainings bei eBuSy gebucht hat.

Warum geeignet:
- hoher operativer Nutzen
- API-nah
- klar begrenzter Scope
- verbessert Arbeitsalltag sofort

# 2. Grundsätzliche Techstack-Entscheidung

## Klare Empfehlung

Die neue Anwendung soll **nicht** als WordPress-Kern gebaut werden.

### WordPress bleibt:
- öffentliche Website
- Content-CMS
- statische / redaktionelle Inhalte

### Die neue App wird:
- eigenständige Anwendung
- auf Subdomain
- mit eigener API
- mit eigener Datenbank
- mit eigenem Frontend

---

## Domain- / Subdomain-Idee

Später angedacht:

- `www.treptower-teufel.de` → öffentliche WordPress-Seite
- `app.treptower-teufel.de` → Mitglieder- und Adminbereich
- `api.treptower-teufel.de` → API für App, Frontend, ggf. WordPress-Widgets

Für Staging später sinngemäß:
- `staging-app...`
- `staging-api...`

---

## Backend

### Entscheidung
**FastAPI**

Warum:
- gut für APIs
- Python-basiert
- saubere Datenmodelle
- gut für Rollen, Rechte, Admin-Workflows
- Integrationen in andere Systeme gut machbar
- geeignet für wachsende Business-Logik

---

## Frontend

Diskutiert wurden:
- SvelteKit
- Svelte + Vite („selbst basteln“)
- WordPress-Plugins

### Ergebnis
**Nicht WordPress als App-Plattform.**

### Kurzfazit
- WordPress-Plugins: nur für kleine Widgets / Darstellungen okay
- Für die eigentliche App ungeeignet als Kern
- Svelte / SvelteKit grundsätzlich geeignet

### Aktuelle Tendenz
Für das eigentliche App-Frontend:
- **Svelte oder SvelteKit**
- Frontend getrennt vom Backend

Es wurde später bewusst offen gelassen, ob für den ersten Schritt eher:
- plain Svelte + Vite
- oder SvelteKit

gebaut wird.

Da Staging erstmal schlicht gehalten werden soll, ist beides noch offen.

---

## Datenbank

### Entscheidung
**PostgreSQL**

Warum:
Die Vereinsdaten sind stark relational:
- Mitglieder
- Status
- Gruppen
- Buchungen
- Trainer
- Arbeitsstunden
- Freigaben
- Rollen
- Rechnungs-/Verwaltungsbezüge

Daher wurde klar festgehalten:
- **SQL ja**
- **PostgreSQL ja**
- **kein NoSQL als Primärdatenbank**

---

## Docker

Es bestehen schlechte Erfahrungen mit Docker.

### Einschätzung
Docker ist **nicht zwingend nötig**.

### Beschluss / Empfehlung
Für dieses Projekt ist ein Start **ohne Docker** völlig legitim.

Stattdessen:
- Ubuntu
- systemd
- FastAPI
- PostgreSQL
- Node
- Reverse Proxy
- Deploy-Skripte

Docker kann später immer noch dazukommen, ist aber kein Muss.

# 3. Hosting-Philosophie

## Europa / deutsche Hand als wichtiges Kriterium

Es wurde ausdrücklich als wichtig festgehalten:
> Das Vereinsprojekt soll möglichst vollständig in europäischer / deutscher Hand liegen.

Darum wurde Hetzner als sehr passend angesehen.

---

## AWS vs. Hetzner

Beruflich existiert bereits ein AWS-basierter Stack:
- EC2
- FastAPI
- Svelte/Vite
- DynamoDB
- SSM für Secrets

### Übertragbarkeit auf Hetzner
Ja, strukturell ist das gut übertragbar, aber **nicht 1:1 als Managed Services**.

### Typische Zuordnung
- EC2 → Hetzner Cloud Server
- DynamoDB → im Vereinsprojekt eher **PostgreSQL**
- SSM → eigene Secrets-/Env-Verwaltung
- S3-artige Assets → optional Hetzner Object Storage

### Wichtige Erkenntnis
Für den Verein ist der Hetzner-Ansatz eher:
- schlicht
- kontrollierbar
- günstig
- gut selbst betreibbar

Nicht:
- AWS-Managed-Services möglichst exakt nachspielen

# 11. Verzeichnisstruktur: aktuelle bewusste Entscheidung

Ursprünglich wurden mehrere konkurrierende Strukturen diskutiert:

* Releases
* Shared
* Apps
* Logs
* komplexere Release-Layouts

Dann wurde bewusst vereinfacht.

## Beschluss für Staging

Für **Staging** wird **kein aufwendiges Release-Modell** verwendet.

Begründung:

* Staging soll erstmal einfach sein
* noch kein echtes Rollback-System nötig
* keine unnötige Komplexität
* Produktivmodell kann später sauberer sein

## Aktuelle Zielstruktur

```text
/srv/tttc
└── app
    ├── api
    │   └── .venv
    ├── env
    ├── logs
    └── web
```

Diese Struktur wurde final für Staging als sinnvoll akzeptiert.

### Bedeutung

* `api` → Backend-Code + Python-venv
* `web` → Frontend-Code
* `env` → serverseitige `.env`-Dateien
* `logs` → Log-Dateien, sofern benötigt

# 12. Deploy-Philosophie

Ein sehr wichtiger Punkt der bisherigen Diskussion:

> App-Code und Projektstruktur sollen später **lokal sauber gebaut** und **per Deploy** auf den Server gebracht werden.

## Konsequenz

Der aktuell auf dem Server liegende Testcode ist **nicht** als finales Projekt-Layout zu verstehen.

### Auf dem Server manuell okay:

* OS
* Pakete
* PostgreSQL
* Python
* Node
* Firewall
* User
* Grundordner

### Im lokalen Git-Repo entstehen später:

* eigentliche Backend-Struktur
* eigentliche Frontend-Struktur
* `pyproject.toml`
* ggf. Lockfiles
* Alembic
* Deploy-Skripte
* systemd-Templates
* Reverse-Proxy-Konfigurationen
* Build- und Startlogik

## Wichtiger Grundsatz

**Server manuell für Infrastruktur.**
**App per Git / Deploy.**

---

# 13. Warum für Staging kein Releases-Modell

Es wurde explizit gefragt, ob man auf Staging überhaupt „lauter releases“ braucht.

## Ergebnis

**Nein.**

### Für Staging reicht:

* ein fester Pfad
* klares Zielverzeichnis
* einfacher Deploy dorthin

### Ein Releases-/Current-/Shared-Modell ist eher sinnvoll für:

* Produktion
* Rollbacks
* mehrere Deploy-Generationen
* stärkere Betriebsähnlichkeit zu Prod

### Daher für Staging bewusst einfach:

```text
/srv/tttc/app/api
/srv/tttc/app/web
/srv/tttc/app/env
/srv/tttc/app/logs
```
